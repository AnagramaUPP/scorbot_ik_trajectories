## Overview

**Scorbot IK Trajectories** is a ROS 2 Jazzy package developed for Cartesian trajectory generation, inverse kinematics validation, and visualization of a Scorbot robotic manipulator in RViz.

The project provides a complete framework for:

* Cartesian trajectory generation.
* Analytical inverse kinematics computation.
* Joint state publishing.
* Robot visualization in RViz.
* Real-time comparison between desired and executed trajectories.
* Validation of inverse kinematics using ROS TF transformations.

The main objective is to verify the consistency between inverse and forward kinematics by comparing the desired Cartesian path against the actual end-effector trajectory obtained from the robot model.

# Inverse Kinematics

The inverse kinematics node receives a desired Cartesian pose:

```text
[x, y, z, delta]
```

where:

* **x** : Cartesian X position
* **y** : Cartesian Y position
* **z** : Cartesian Z position
* **delta** : Wrist orientation

The algorithm computes the corresponding joint angles:

```text
θ1
θ2
θ3
θ4
θ5
```

which are published through:

```text
/joint_states
```

for visualization and kinematic validation.

# Cartesian Trajectory Planning

The package currently implements a **Lemniscate (∞) trajectory**.

The trajectory can be generated in different Cartesian planes.

## XY Plane

```math
x = x_0 + A\sin(t)
```

```math
y = y_0 + B\sin(t)\cos(t)
```

```math
z = z_0
```

# Inverse Kinematics Validation

Two RViz markers are used to validate the inverse kinematics solution.

## Desired Trajectory

Topic:

```text
/scorbot/trajectory_marker
```

Color:

```text
Red
```

This marker represents the reference Cartesian trajectory generated by the trajectory planner.

---

## Actual Trajectory

Topic:

```text
/scorbot/real_trajectory_marker
```

Color:

```text
Blue
```

This marker represents the actual trajectory of the end-effector obtained from the TF transformation:

```text
base_link → link_5
```

computed by:

```text
robot_state_publisher
```

---

# Validation Methodology

The inverse kinematics solution is validated by comparing:

```text
Desired Cartesian Trajectory
```

against

```text
Actual End-Effector Trajectory
```

When both trajectories overlap, the consistency between:

```text
Desired Cartesian Pose
        ↓
Inverse Kinematics
        ↓
Joint States
        ↓
Forward Kinematics (TF)
```

is verified.

This approach allows the identification of:

* Workspace limitations.
* Singular configurations.
* Kinematic inconsistencies.
* Cartesian tracking errors.
* Reachability constraints.

# Running the Package

## Build

```bash
cd ~/ros2_ws

colcon build --packages-select scorbot_ik_trajectories

source install/setup.bash
```

---

## Launch RViz

```bash
ros2 launch scorbot_ik_trajectories display.launch.py
```

---

## Run the Inverse Kinematics Node

```bash
ros2 run scorbot_ik_trajectories scorbot_ik_publisher.py
```

---

## Run the Trajectory Generator

```bash
ros2 run scorbot_ik_trajectories scorbot_trajectory_publisher.py
```
